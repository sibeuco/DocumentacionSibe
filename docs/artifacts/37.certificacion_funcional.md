## 1. Introducción y Alcance

### 1.1 Propósito

Este artefacto constituye el **informe formal de certificación funcional** del sistema SIBE. Su objetivo es emitir un veredicto técnico documentado sobre si el sistema, tal y como existe en la fecha de evaluación, cumple con los criterios de calidad requeridos para su operación en un ambiente de producción académica.

La certificación funcional es el proceso mediante el cual se comprueba objetivamente que:

1. Los casos de prueba definidos en el Artefacto 35 se ejecutan y producen resultados verificables.
2. La cobertura de código alcanza los umbrales establecidos por los estándares de la industria.
3. Los requisitos funcionales derivados de las historias de usuario (Artefacto 16) están validados por pruebas automatizadas.
4. No existen vulnerabilidades de seguridad críticas sin mitigar en el código fuente y la configuración de la aplicación.
5. Los riesgos residuales identificados son documentados y aceptados o escalados para remediación.

### 1.2 Alcance de la Certificación

| Componente | Versión evaluada | Ambiente | Incluido |
|-----------|-----------------|----------|----------|
| Backend — Spring Boot API | Spring Boot 3.5.0 / Java 17 | Pruebas locales (H2 in-memory) | ✅ |
| Frontend — Angular SPA | Angular 16.2 / TypeScript 5.1 | Pruebas unitarias (ChromeHeadless) | ✅ |
| Base de datos PostgreSQL | `sibe_db2` @ localhost:5432 | Solo integración transaccional | ⚠️ Parcial |

### 1.4 Metodología de Evaluación

La certificación se produjo mediante los siguientes pasos:

1. **Análisis estático del código fuente** — Lectura directa de los archivos fuente backend (Java 17) y frontend (TypeScript 5.1) para validar estructura, seguridad y completitud.
2. **Revisión de resultados de pruebas** — Lectura de los archivos XML generados por JUnit 5 en `build/test-results/` y revisión del reporte HTML de Karma.
3. **Revisión de informes de cobertura** — Lectura del reporte JaCoCo HTML/CSV (`build/jacocoHtml/`) y del reporte Istanbul (`coverage/sibe-frontend/`).
4. **Trazabilidad** — Cruzamiento de los casos de prueba del Artefacto 35 con las historias de usuario del Artefacto 16.
5. **Auditoría de seguridad** — Revisión de configuración de seguridad Spring, JWT, CORS y propiedades de la aplicación contra OWASP Top 10.

## 2. Criterios de Certificación

### 2.1 Umbrales de Cobertura de Código

Los siguientes umbrales son adoptados como requisitos mínimos para la emisión de una certificación favorable:

| Nivel | Instrucciones / Stmts | Ramas / Branches | Funciones | Líneas | Descripción |
|-------|----------------------|-----------------|-----------|--------|-------------|
| **Excelente** | ≥ 90% | ≥ 85% | ≥ 90% | ≥ 90% | Cobertura de referencia para sistemas críticos |
| **✅ Aceptable (umbral mínimo)** | ≥ 80% | ≥ 70% | ≥ 80% | ≥ 80% | Quality Gate SonarQube por defecto |
| **⚠️ Mínimo con deuda técnica** | ≥ 70% | ≥ 60% | ≥ 70% | ≥ 70% | Aceptable con plan de mejora |
| **❌ Insuficiente** | ≥ 50% | ≥ 40% | — | — | No certifiable sin remediación |
| **🔴 Crítico** | < 50% | < 40% | — | — | No certifiable — riesgo de regresión severo |

### 2.2 Requisitos de Suite de Pruebas

| Criterio | Umbral | Valor para certificación APROBADA |
|----------|--------|-----------------------------------|
| Tests que pasan (pass rate) | 100% | 0 fallos permitidos |
| Tests con error estructural | 0% | Ningún test en estado ERROR |
| Tests omitidos (skipped) | < 5% | Omisión justificada documentada |
| Clases con 0 tests activos | 0 | Toda clase bajo prueba debe tener ≥ 1 test ejecutable |

### 2.3 Requisitos de Seguridad

Para certificar el componente de seguridad:

| Categoría OWASP | Criterio | Severidad si no se cumple |
|-----------------|----------|--------------------------|
| A01 — Broken Access Control | Todos los endpoints protegidos tienen `@PreAuthorize` | 🔴 Bloqueante |
| A02 — Cryptographic Failures | Claves y secretos no están en código fuente | 🔴 Bloqueante |
| A03 — Injection | Uso de JPA parametrizado (no SQL concatenado) | 🔴 Bloqueante |
| A05 — Security Misconfiguration | No hay credenciales por defecto en producción | 🔴 Bloqueante |
| A07 — Identification Failures | JWT válido y expiración adecuada | 🟡 Advertencia |
| A09 — Logging and Monitoring | Logging de login y autenticación implementado | 🟡 Advertencia |

### 2.4 Requisitos Funcionales Mínimos

Para certificar la funcionalidad del sistema, cada historia de usuario del Artefacto 16 debe tener al menos **un caso de prueba positivo ejecutable** que valide su flujo principal (happy path).

### 4.3 Análisis de Causa Raíz — Tests No Ejecutados

**Situación observada:** El Artefacto 32 documenta 96 clases de prueba. El Artefacto 34 y la revisión directa del directorio `build/test-results/` confirman que solo 2 clases producen resultados de ejecución. Las 94 clases restantes no se ejecutan.

**Hipótesis de causa raíz identificadas (por orden de probabilidad):**

#### Causa 1 — ALTA probabilidad: Patrón `@Mock` sobre la clase bajo prueba

Las 94 clases de prueba no detectadas por Gradle siguen este patrón estructural incorrecto:

```java
// Patrón real encontrado en las clases de prueba (NO ejecuta código de producción)
@ExtendWith(MockitoExtension.class)
class GuardarActividadManejadorTest {

    @Mock
    private GuardarActividadManejador manejador;   // ← La propia clase bajo prueba está mockeada

    @Test
    void deberiaEjecutarGuardarActividad() {
        var idEsperado = UUID.randomUUID();
        when(manejador.ejecutar(any())).thenReturn(idEsperado);  // ← Se stubea el mock
        var respuesta = manejador.ejecutar(new ComandoGuardarActividad());
        assertEquals(idEsperado, respuesta);  // ← Se verifica el mock, no el código real
    }
}
```

En este patrón, **Gradle SÍ descubre y ejecuta la prueba** (consta como passed), pero JaCoCo registra 0% porque el código de producción de `GuardarActividadManejador` no es invocado — el mock intercepta toda llamada. Sin embargo, si Gradle no genera el XML de resultado, puede ser que las clases no siquiera sean encontradas por el classpath.

#### Causa 2 — MEDIA probabilidad: Errores de compilación silenciosos en los 94 test classes

Si las 94 clases tienen errores de importación, dependencias no resueltas o problemas de configuración del classpath de `testImplementation`, Gradle las excluiría del ciclo de prueba sin reportar error (cuando `ignoreFailures = true`).

#### Causa 3 — MEDIA probabilidad: Convención de nombres no reconocida por JUnit Platform

JUnit 5 Platform descubre pruebas por convención. Si los nombres de clase no terminan en `Test` o si los métodos carecen de `@Test`, no son descubiertos. Las 2 clases que sí se ejecutan terminan en `Test`. Las 94 restantes también parecen seguir la convención, pero podría haber variaciones.

#### Causa 4 — BAJA probabilidad: Los archivos XML de resultados existen pero están en un directorio diferente

Es posible que los 94 test class XML estén en un subdirectorio de `build/test-results/` no revisado por los subagentes. Esta causa requeriría una inspección exhaustiva del sistema de archivos.

**Evidencia JaCoCo que respalda las causas 1 y 2:**

Si las pruebas se ejecutaran correctamente (incluso con mocks), JaCoCo registraría al menos la instanciación de las clases bajo prueba cuando se usan `@InjectMocks`. La cobertura de exactamente 0% en todos esos paquetes confirma que **ningún código de producción en esas capas fue ejecutado en ningún momento**.

## 7. Hallazgos de Seguridad (OWASP)

Se identificaron 7 hallazgos de seguridad durante la revisión del código fuente y la configuración de la aplicación. Están clasificados por categoría OWASP Top 10 (2021) y severidad.

### 7.1 Hallazgo SEC-001 — Secreto JWT Hardcodeado en Código Fuente

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-001 |
| **Categoría OWASP** | A02:2021 — Cryptographic Failures |
| **Severidad** | 🔴 CRÍTICA |
| **Archivo** | `SeguridadConstante.java` (infra/seguridad) |
| **Descripción** | La clave de firmado de JWT (`JWT_KEY`) es un string literal hardcodeado: `"jxgEQeXHuPq8VdbyYFNkANdudQ53YUn4"`. Esta clave se usa para firmar y verificar **todos** los tokens JWT del sistema. Si este archivo se expone en un repositorio público, cualquier atacante puede generar tokens JWT válidos para cualquier usuario. |
| **Impacto** | Compromiso total de la autenticación del sistema. Un atacante puede generar tokens válidos para cualquier rol (`ADMINISTRADOR_DIRECCION`). |
| **Remediación** | Externalizar la clave a una variable de entorno: `@Value("${jwt.secret}")` + configurar `JWT_SECRET=...` en el servidor. Rotar la clave actual inmediatamente. |
| **Bloqueante para cert.** | **SÍ** |

---

### 7.2 Hallazgo SEC-002 — Contraseña de Base de Datos en Texto Claro

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-002 |
| **Categoría OWASP** | A05:2021 — Security Misconfiguration |
| **Severidad** | 🔴 ALTA |
| **Archivo** | `application.properties` |
| **Descripción** | `spring.datasource.password=admin1234` en texto claro. Si el repositorio es privado —pero accesible a desarrolladores no autorizados— esta credencial expone la base de datos PostgreSQL. |
| **Impacto** | Acceso directo a la base de datos del sistema, incluyendo datos de usuarios, actividades y participantes. |
| **Remediación** | Reemplazar con variable de entorno: `spring.datasource.password=${DB_PASSWORD}`. Crear perfiles `application-dev.properties` y `application-prod.properties`. |
| **Bloqueante para cert.** | **SÍ** |

---

### 7.3 Hallazgo SEC-003 — App Password de Correo en Texto Claro

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-003 |
| **Categoría OWASP** | A05:2021 — Security Misconfiguration |
| **Severidad** | 🔴 ALTA |
| **Archivo** | `application.properties` |
| **Descripción** | `spring.mail.password=atsiysuskgnqccrk` (app password de Gmail) en texto claro. Expone la cuenta de correo del sistema a uso malicioso (spam, phishing) y puede comprometer la cuenta institucional. |
| **Remediación** | Externalizar a variable de entorno: `spring.mail.password=${MAIL_APP_PASSWORD}`. Revocar y generar nuevo app password. |
| **Bloqueante para cert.** | **SÍ** |

---

### 7.4 Hallazgo SEC-004 — CORS Restringido Solo a Localhost

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-004 |
| **Categoría OWASP** | A05:2021 — Security Misconfiguration |
| **Severidad** | 🟡 MEDIA (solo afecta producción) |
| **Descripción** | La configuración CORS permite solo `http://localhost:4200`. En despliegue de producción, el origen debe actualizarse al dominio real del frontend. De lo contrario, el frontend de producción no podrá llamar a la API. |
| **Remediación** | `@Value("${cors.allowed-origin}")` para hacer el origen configurable por ambiente. |
| **Bloqueante para cert.** | No (para ambiente de desarrollo). Sí para producción. |

---

### 7.5 Hallazgo SEC-005 — `spring.jpa.hibernate.ddl-auto=update` en Producción

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-005 |
| **Categoría OWASP** | A05:2021 — Security Misconfiguration |
| **Severidad** | 🟡 MEDIA |
| **Descripción** | `ddl-auto=update` permite a Hibernate alterar el esquema de la base de datos en tiempo de ejecución. En producción, esto puede resultar en pérdida de datos si se cambia el modelo de dominio. |
| **Remediación** | Usar `ddl-auto=validate` o `none` en producción. Gestionar el esquema con Flyway o Liquibase. |
| **Bloqueante para cert.** | No para desarrollo/staging. Sí para producción. |

---

### 7.6 Hallazgo SEC-006 — Sin Separación de Perfiles por Ambiente

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-006 |
| **Categoría OWASP** | A05:2021 — Security Misconfiguration |
| **Severidad** | 🟡 MEDIA |
| **Descripción** | Un único archivo `application.properties` sirve todos los ambientes (dev, test, prod). Esto impide aplicar configuraciones de seguridad diferenciales por ambiente. Por ejemplo, `ignoreFailures=true` es aceptable en dev pero no en un pipeline CI/CD de producción. |
| **Remediación** | Crear `application-dev.properties`, `application-prod.properties`. Usar `spring.profiles.active=${ENVIRONMENT}`. |
| **Bloqueante para cert.** | No para desarrollo actual. |

---

### 7.7 Hallazgo SEC-007 — Expiración JWT de 8.3 Horas (Potencialmente Larga)

| Campo | Detalle |
|-------|---------|
| **ID** | SEC-007 |
| **Categoría OWASP** | A07:2021 — Identification and Authentication Failures |
| **Severidad** | 🟢 BAJA |
| **Descripción** | El JWT expira en 30 000 000 milisegundos (~8.3 horas). Una sesión de 8 horas es razonable para uso académico durante, pero no implementa refresh tokens. Si el JWT es robado, el atacante tiene ~8 horas de ventana de uso. |
| **Remediación** | Considerar implementar refresh tokens para reducir la expiración del access token a 15-30 minutos. |
| **Bloqueante para cert.** | No |
