## 1. Introducción y Propósito

### 1.1 Propósito

Este artefacto describe el **plan completo de despliegue y puesta en operación** del sistema SIBE. Cubre los procedimientos técnicos necesarios para llevar el sistema desde el código fuente hasta un ambiente de producción funcional y seguro, incluyendo configuración de infraestructura, builds reproducibles, inicialización de datos, estrategia de rollback y operación continua.

El plan aplica a los siguientes ambientes:

| Ambiente | Propósito | Audiencia |
|----------|-----------|-----------|
| **Desarrollo (dev)** | Trabajo diario del equipo de desarrollo | Desarrolladores |
| **Pruebas (staging)** | Validación funcional antes del release | QA, Product Owner |
| **Producción (prod)** | Servicio real a la UCO | Usuarios finales, administradores |

### 1.2 Alcance

Este plan cubre:
- Build y empaquetado del backend (Spring Boot → JAR ejecutable).
- Build y optimización del frontend (Angular → assets estáticos).
- Configuración de PostgreSQL (esquema, datos semilla, usuarios).
- Configuración del servidor web nginx para servir la SPA.
- Gestión de secretos y variables de entorno.
- Configuración Docker / docker-compose (arquetipo recomendado).
- Procedimiento de despliegue manual alternativo.
- Verificación post-despliegue (smoke tests).
- Estrategia de rollback.

### 1.3 Restricciones Conocidas (del Artefacto 36)

> ⚠️ Antes de desplegar en producción, los siguientes puntos del Artefacto 36 (Certificación Funcional) deben estar resueltos:
>
> - **SEC-001:** Secreto JWT no debe estar hardcodeado — configurar mediante variable de entorno.
> - **SEC-002:** Contraseña de BD no debe estar en `application.properties` en texto claro.
> - **SEC-003:** App password de correo no debe estar en texto claro en el repositorio.
>
> Este plan incorpora las prácticas de seguridad correctas para todos los ambientes desde la sección §5.

## 3. Arquitectura de Despliegue

### 3.1 Diagrama de Componentes

```
┌─────────────────────────────────────────────────────────────────────┐
│                        SERVIDOR DE PRODUCCIÓN                        │
│                                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐ │
│  │                          nginx (puerto 80/443)                   │ │
│  │                                                                   │ │
│  │   /              → /var/www/sibe-frontend/     (Angular SPA)     │ │
│  │   /api/*         → http://localhost:8080/api/* (Spring Boot API) │ │
│  └────────────────────────────────────────────────────────────────-─┘ │
│           │                              │                             │
│           ▼                              ▼                             │
│  ┌──────────────────┐         ┌─────────────────────┐                │
│  │  Angular SPA     │         │  Spring Boot API     │                │
│  │  (archivos est.) │         │  sibe-0.0.1.jar      │                │
│  │  dist/sibe-      │         │  puerto 8080         │                │
│  │  frontend/       │         │  contexto: /api      │                │
│  └──────────────────┘         └──────────┬──────────┘                │
│                                           │                            │
│                                ┌──────────▼──────────┐               │
│                                │   PostgreSQL         │               │
│                                │   sibe_db2           │               │
│                                │   puerto 5432        │               │
│                                └─────────────────────┘               │
│                                           │                            │
│                               ┌───────────▼─────────────┐            │
│                               │    Gmail SMTP (externo)  │            │
│                               │    smtp.gmail.com:587    │            │
│                               └─────────────────────────┘            │
└─────────────────────────────────────────────────────────────────────┘

CLIENTE NAVEGADOR
    │
    └── HTTPS → nginx (80/443) → Angular SPA o Spring Boot API
```

### 3.2 Flujo de Petición HTTP

```
[Navegador]
    │
    ├── GET /            → nginx sirve index.html (SPA routing)
    ├── GET /assets/*    → nginx sirve archivos estáticos optimizados
    └── * /api/*         → nginx hace proxy_pass a Spring Boot :8080
                              │
                              ├── Filtros de Seguridad JWT
                              ├── Controladore REST
                              ├── Capa de Aplicación (CQRS)
                              ├── Dominio (Hexagonal)
                              └── JPA → PostgreSQL
```

### 3.3 Puertos y Servicios

| Servicio | Puerto interno | Puerto externo | Protocolo |
|---------|---------------|---------------|-----------|
| nginx | 80 | 80 | HTTP |
| nginx | 443 | 443 | HTTPS (configurar certificado TLS) |
| Spring Boot API | 8080 | Solo via nginx | HTTP interno |
| PostgreSQL | 5432 | NO exponer | TCP |

> **Regla de seguridad:** PostgreSQL y el puerto 8080 de Spring Boot **nunca deben estar expuestos directamente** a internet. Todo el tráfico externo debe pasar por nginx.

## 5. Configuración del Entorno

### 5.1 Variables de Entorno Requeridas

Antes de ejecutar la aplicación, deben configurarse las siguientes variables de entorno en el sistema operativo del servidor (o en el archivo `.env` si se usa Docker Compose):

#### 5.1.1 Variables del Backend

| Variable | Descripción | Ejemplo |
|----------|-------------|---------|
| `DB_HOST` | Host de PostgreSQL | `localhost` |
| `DB_PORT` | Puerto de PostgreSQL | `5432` |
| `DB_NAME` | Nombre de la base de datos | `sibe_db2` |
| `DB_USERNAME` | Usuario de PostgreSQL | `sibe_user` |
| `DB_PASSWORD` | Contraseña del usuario PostgreSQL | *(secreto — no poner aquí)* |
| `JWT_SECRET` | Clave secreta para firmar JWT (mínimo 256 bits) | *(secreto — generar aleatoriamente)* |
| `MAIL_HOST` | Host del servidor SMTP | `smtp.gmail.com` |
| `MAIL_PORT` | Puerto SMTP | `587` |
| `MAIL_USERNAME` | Correo emisor | `sibeapplicationuco@gmail.com` |
| `MAIL_PASSWORD` | App password del correo | *(secreto — no poner aquí)* |
| `CORS_ALLOWED_ORIGIN` | URL del frontend en producción | `https://sibe.uco.edu.co` |
| `SPRING_PROFILES_ACTIVE` | Perfil activo | `prod` |
| `SERVER_PORT` | Puerto del servidor (opcional, default 8080) | `8080` |

#### 5.1.2 Generar el Secreto JWT

El secreto JWT debe tener al menos 256 bits (32 bytes) pseudoaleatorios. Generar con:

```bash
# Linux / macOS
openssl rand -hex 32

# PowerShell (Windows)
[System.Convert]::ToBase64String([System.Security.Cryptography.RandomNumberGenerator]::GetBytes(32))

# Ejemplo de salida: a7f3c2d1e8b4f9a6c3d2e1f8b4a7c3d2e1f8b4a7c3d2e1f8b4a7c3d2e1f8b4a7
```

#### 5.1.3 Configurar Variables en el Sistema

**Linux / systemd (recomendado para producción):**

```bash
# Crear archivo de variables de entorno
sudo tee /etc/sibe/sibe.env > /dev/null <<EOF
DB_HOST=localhost
DB_PORT=5432
DB_NAME=sibe_db2
DB_USERNAME=sibe_user
DB_PASSWORD=<contraseña_segura_aquí>
JWT_SECRET=<secreto_jwt_256_bits_aquí>
MAIL_HOST=smtp.gmail.com
MAIL_PORT=587
MAIL_USERNAME=sibeapplicationuco@gmail.com
MAIL_PASSWORD=<app_password_gmail_aquí>
CORS_ALLOWED_ORIGIN=https://sibe.uco.edu.co
SPRING_PROFILES_ACTIVE=prod
EOF

# Permisos restringidos (solo root puede leer)
sudo chmod 600 /etc/sibe/sibe.env
sudo chown root:root /etc/sibe/sibe.env
```

**Windows (PowerShell — para entorno de desarrollo/staging):**

```powershell
[System.Environment]::SetEnvironmentVariable("DB_PASSWORD", "valor", "Machine")
[System.Environment]::SetEnvironmentVariable("JWT_SECRET", "valor", "Machine")
[System.Environment]::SetEnvironmentVariable("MAIL_PASSWORD", "valor", "Machine")
```

### 5.2 Configuración de Base de Datos PostgreSQL

#### 5.2.1 Crear usuario y base de datos

```sql
-- Ejecutar como superusuario postgres
CREATE USER sibe_user WITH PASSWORD 'contraseña_segura_aqui';

CREATE DATABASE sibe_db2
    WITH OWNER = sibe_user
    ENCODING = 'UTF8'
    LC_COLLATE = 'es_CO.UTF-8'
    LC_CTYPE = 'es_CO.UTF-8'
    TEMPLATE = template0;

-- Otorgar permisos
GRANT ALL PRIVILEGES ON DATABASE sibe_db2 TO sibe_user;
\c sibe_db2
GRANT ALL ON SCHEMA public TO sibe_user;
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA public TO sibe_user;
GRANT ALL PRIVILEGES ON ALL SEQUENCES IN SCHEMA public TO sibe_user;
```

> **Nota:** El aplicativo usa `spring.jpa.hibernate.ddl-auto=update` en el perfil de desarrollo, lo que hace que Hibernate cree/altere el esquema automáticamente al iniciar. En producción se recomienda usar `validate` (ver §15).

#### 5.2.2 Configuración de pg_hba.conf (para conexiones locales)

```conf
# /etc/postgresql/16/main/pg_hba.conf
# Agregar línea para usuario sibe_user (conexión local via contraseña)
local   sibe_db2        sibe_user                  md5
host    sibe_db2        sibe_user   127.0.0.1/32   md5
host    sibe_db2        sibe_user   ::1/128         md5
```

```bash
sudo systemctl reload postgresql
```

#### 5.2.3 Verificar conexión

```bash
psql -U sibe_user -d sibe_db2 -h localhost -c "SELECT version();"
```

### 5.3 Configuración del Servidor SMTP (Correo)

El sistema usa Gmail via SMTP con STARTTLS para envío de correos de recuperación de contraseña.

**Requisito:** Una cuenta de Gmail con **App Password** habilitada (no la contraseña normal de la cuenta). Para generar un App Password:

1. Ir a la cuenta de Google → Seguridad → Verificación en 2 pasos (debe estar activa).
2. En "Contraseñas de aplicación" → seleccionar "Correo" y "Otro" → generar.
3. Copiar el código de 16 caracteres → asignarlo a la variable `MAIL_PASSWORD`.

```properties
# Propiedades SMTP configuradas en application.properties
spring.mail.host=smtp.gmail.com
spring.mail.port=587
spring.mail.properties.mail.smtp.auth=true
spring.mail.properties.mail.smtp.starttls.enable=true
```

> Todos los valores sensibles (`spring.mail.username`, `spring.mail.password`) deben venir de variables de entorno, nunca de `application.properties` en el repositorio.

### 5.4 Configuración de Seguridad JWT

La clase `SeguridadConstante` en el backend debe leer la clave JWT de la variable de entorno:

**Estado actual (inseguro — para referencia):**
```java
// SeguridadConstante.java — ESTADO ACTUAL (NO apto para producción)
public static final String JWT_KEY = "jxgEQeXHuPq8VdbyYFNkANdudQ53YUn4";
```

**Estado recomendado (antes del despliegue en producción):**
```java
// SeguridadConstante.java — versión segura
@Value("${jwt.secret}")
private String jwtSecret;
```

**`application-prod.properties`:**
```properties
jwt.secret=${JWT_SECRET}
```

### 6.4 Estrategia de inicio con DataLoaders

Al arrancar por primera vez, Spring Boot ejecuta automáticamente los `CommandLineRunner` (`DataLoader`) en el siguiente **orden** (controlado por `@Order`):

| Orden | DataLoader | Datos cargados | Condición |
|-------|-----------|----------------|-----------|
| **1** | `TipoUsuarioDataLoader` | 3 tipos de usuario: `ADMINISTRADOR_DIRECCION`, `ADMINISTRADOR_AREA`, `COLABORADOR` | Solo si no existen tipos de usuario |
| **2** | `TipoIdentificacionDataLoader` | 3 tipos: CC (Cédula Ciudadanía), TI (Tarjeta Identidad), CE (Cédula Extranjería) | Solo si no existen |
| **3** | `TipoIndicadorDataLoader` | 5+ tipos de indicador: Eficiencia, Capacidad Instalada, Eficacia, Efectividad, Valor | Solo si no existen |
| **4** | `EstadoActividadDataLoader` | 3 estados: `PENDIENTE`, `EN_CURSO`, `FINALIZADA` | Solo si no existen |
| **5** | `TemporalidadDataLoader` | 5 temporalidades: Diaria, Semanal, Mensual, Trimestral, Anual | Solo si no existen |
| **6** | `PublicoInteresDataLoader` | 5 públicos de interés (Registros Calificados, Acreditación, ISO, Ministerio, Plan Pastoral) | Solo si no existen |
| **7** | `SubareaDataLoader` | 8 subáreas: Deportes, Cancha, Extensión Cultural, Banda Sinfónica, Gimnasio, Unidad Salud, Acompañamiento, Trabajo Social | Solo si no existen |
| **8** | `AreaDataLoader` | 4 áreas: Bienestar (con 8 subáreas), Evangelización, Hogar Juvenil, Servicio de Atención | Solo si no existen |
| **9** | `DireccionDataLoader` | 1 dirección: `BIENESTAR ESTUDIANTIL Y EVANGELIZACION` (con las 4 áreas) | Solo si no existen |
| **10** | `UsuarioAdministradorDataLoader` | 1 usuario administrador UCO por defecto | Solo si no existen usuarios |

> **Comportamiento idempotente:** Cada DataLoader verifica con `debenCargarseDatos()` si los datos ya existen antes de insertarlos. El mecanismo es seguro para re-ejecuciones: si la aplicación se reinicia, no duplica datos.

> ⚠️ **Credenciales del administrador por defecto:** El usuario administrador inicial se crea con las constantes definidas en `DatoConstante`. Se debe cambiar la contraseña del administrador inmediatamente después del primer despliegue.

### 7.3 Configuración de nginx para SPA

El frontend Angular usa **rutas del lado del cliente** (HTML5 History API). nginx debe redirigir todas las rutas que no sean archivos estáticos o `/api/` a `index.html`:

```nginx
# /etc/nginx/sites-available/sibe
server {
    listen 80;
    listen [::]:80;
    server_name sibe.uco.edu.co www.sibe.uco.edu.co;

    # Redirigir HTTP a HTTPS en producción
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name sibe.uco.edu.co www.sibe.uco.edu.co;

    # Certificados TLS (Let's Encrypt o certificado UCO)
    ssl_certificate     /etc/nginx/ssl/sibe.uco.edu.co.crt;
    ssl_certificate_key /etc/nginx/ssl/sibe.uco.edu.co.key;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    # Servir Angular SPA
    root /var/www/sibe-frontend;
    index index.html;

    # Archivos estáticos — caché agresiva (tienen hash en nombre)
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
        try_files $uri =404;
    }

    # Angular SPA — todas las rutas no-API y no-static → index.html
    location / {
        try_files $uri $uri/ /index.html;
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # Reverse proxy hacia Spring Boot API
    location /api/ {
        proxy_pass         http://127.0.0.1:8080;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;

        # Timeouts apropiados para operaciones largas (carga masiva Excel)
        proxy_connect_timeout  60s;
        proxy_send_timeout    300s;
        proxy_read_timeout    300s;

        # Tamaño máximo de cuerpo de petición (para carga de archivos Excel)
        client_max_body_size  50M;
    }

    # Logs
    access_log /var/log/nginx/sibe-access.log;
    error_log  /var/log/nginx/sibe-error.log warn;
}
```

## 13. Estrategia de Rollback

### 13.1 Rollback del Backend

```bash
# 1. Detener el servicio actual
sudo systemctl stop sibe-backend

# 2. Reemplazar el JAR por la versión anterior
# (mantener siempre el JAR de la versión anterior en /opt/sibe/backend/backup/)
sudo mkdir -p /opt/sibe/backend/backup
sudo cp /opt/sibe/backend/sibe-0.0.1-SNAPSHOT.jar /opt/sibe/backend/backup/sibe-$(date +%Y%m%d_%H%M%S).jar
sudo cp /opt/sibe/backend/backup/sibe-VERSION_ANTERIOR.jar /opt/sibe/backend/sibe-0.0.1-SNAPSHOT.jar

# 3. Reiniciar servicio
sudo systemctl start sibe-backend
sudo systemctl status sibe-backend
```

### 13.2 Rollback del Frontend

```bash
# 1. Mantener backup del build anterior
sudo cp -r /var/www/sibe-frontend /var/www/sibe-frontend-backup-$(date +%Y%m%d_%H%M%S)

# 2. Restaurar versión anterior
sudo rm -rf /var/www/sibe-frontend/*
sudo cp -r /var/www/sibe-frontend-backup-VERSION_ANTERIOR/* /var/www/sibe-frontend/
sudo chown -R www-data:www-data /var/www/sibe-frontend
sudo nginx -s reload
```

### 13.3 Rollback con Docker Compose

```bash
# Revertir imagen del backend
docker compose stop backend
docker tag sibe-backend:current sibe-backend:rollback-failed
docker tag sibe-backend:previous sibe-backend:current
docker compose up -d backend

# Verificar
docker compose ps
docker compose logs backend
```

### 13.4 Rollback de Base de Datos

> ⚠️ **El rollback de base de datos es el más riesgoso.** El esquema lo maneja `ddl-auto=update`, lo que significa que el rollback de columnas/tablas añadidas en una nueva versión requiere scripts SQL manuales.

**Buena práctica antes de desplegar:**

```bash
# 1. Hacer backup de la base de datos ANTES del despliegue
sudo -u postgres pg_dump sibe_db2 > /opt/sibe/backup/sibe_db2_$(date +%Y%m%d_%H%M%S).sql

# 2. Si hay rollback
sudo systemctl stop sibe-backend
sudo -u postgres psql -c "DROP DATABASE IF EXISTS sibe_db2;"
sudo -u postgres psql -c "CREATE DATABASE sibe_db2 WITH OWNER sibe_user ENCODING 'UTF8';"
sudo -u postgres psql sibe_db2 < /opt/sibe/backup/sibe_db2_FECHA_ANTERIOR.sql
sudo systemctl start sibe-backend
```

### 14.2 Monitoreo de Salud

**Script de health check básico (`/opt/sibe/health-check.sh`):**

```bash
#!/bin/bash
# Health check del sistema SIBE

BACKEND_URL="http://localhost:8080/api"
FRONTEND_URL="http://localhost"
LOG_FILE="/var/log/sibe-health.log"
DATE=$(date '+%Y-%m-%d %H:%M:%S')

# Check backend
BACKEND_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$BACKEND_URL/tipos_usuario")
if [ "$BACKEND_STATUS" != "200" ]; then
    echo "[$DATE] ERROR: Backend no responde (HTTP $BACKEND_STATUS)" >> "$LOG_FILE"
    # Aquí se podría agregar: envío de email de alerta, reinicio automático, etc.
fi

# Check frontend
FRONTEND_STATUS=$(curl -s -o /dev/null -w "%{http_code}" "$FRONTEND_URL/")
if [ "$FRONTEND_STATUS" != "200" ]; then
    echo "[$DATE] ERROR: Frontend no responde (HTTP $FRONTEND_STATUS)" >> "$LOG_FILE"
fi

# Check PostgreSQL
if ! sudo -u postgres pg_isready -q; then
    echo "[$DATE] ERROR: PostgreSQL no está disponible" >> "$LOG_FILE"
fi

echo "[$DATE] Health check OK — Backend: $BACKEND_STATUS, Frontend: $FRONTEND_STATUS" >> "$LOG_FILE"
```

```bash
# Ejecutar cada 5 minutos via cron
echo "*/5 * * * * root /opt/sibe/health-check.sh" | sudo tee /etc/cron.d/sibe-health
```

### 14.3 Gestión de Sesiones JWT

- Los tokens JWT expiran en **~8.3 horas** (30 000 000 ms).
- La aplicación no implementa refresh tokens — los usuarios deben hacer login nuevamente tras la expiración.
- No hay mecanismo de revocación de tokens. Si un token es comprometido, debe rotarse el secreto JWT (lo que invalida **todos** los tokens activos en el sistema).

### 14.4 Respaldos Programados

```bash

0 2 * * * postgres pg_dump sibe_db2 > /opt/sibe/backup/sibe_db2_$(date +\%Y\%m\%d).sql
# Mantener solo los últimos 30 días de backups
0 3 * * * find /opt/sibe/backup -name "sibe_db2_*.sql" -mtime +30 -delete
EOF
```
