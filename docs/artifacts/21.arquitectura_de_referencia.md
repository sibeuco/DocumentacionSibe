# 21. Arquitectura de Referencia

## 1. Propósito y Alcance

Este documento describe la arquitectura de referencia del sistema SIBE (Sistema de Información de Bienestar Estudiantil), sirviendo como guía técnica autoritativa para comprender, mantener y evolucionar la solución. Está construido a partir del análisis exhaustivo del código fuente (backend y frontend) y de los artefactos de documentación del proyecto.

**Audiencia:** Arquitectos de software, desarrolladores, equipo de operaciones, auditores técnicos y stakeholders con interés en la estructura técnica del sistema.

**Alcance:** Cubre las vistas arquitectónicas (contexto, contenedores, componentes, datos, seguridad, despliegue), las decisiones arquitectónicas implementadas, los atributos de calidad medibles, las restricciones técnicas y los riesgos arquitectónicos identificados.

## 2. Contexto del Sistema

### 2.1 Descripción General

SIBE es una aplicación web diseñada para digitalizar la gestión de actividades y asistencia de la Dirección de Bienestar y Evangelización de la Universidad Católica de Oriente (UCO). Reemplaza procesos manuales basados en planillas de papel, Google Forms, CheckPoint y AccuClass.

### 5.2 CQRS — Separación Comando/Consulta

SIBE implementa **CQRS estructural** en todas las capas. La separación no es eventual (sin event sourcing) sino paquetizada: las rutas de escritura y lectura son paquetes independientes en cada capa.

```mermaid
flowchart LR
    subgraph Escritura ["Escritura (Comando)"]
        direction TB
        CC[ComandoControlador] --> MC["ManejadorComandoRespuesta\n@Transactional"]
        MC --> UC["GuardarXUseCase /\nModificarXUseCase /\nEliminarXUseCase"]
        UC --> PC["RepositorioComando\n(puerto)"]
        PC --> IC[RepositorioComandoImpl]
    end

    subgraph Lectura ["Lectura (Consulta)"]
        direction TB
        CQ[ConsultaControlador] --> MQ["ManejadorRespuesta /\nManejadorParametroRespuesta\n@Transactional readOnly"]
        MQ --> UQ["ConsultarXUseCase /\nListarXUseCase"]
        UQ --> PQ["RepositorioConsulta\n(puerto)"]
        PQ --> IQ[RepositorioConsultaImpl]
    end

    IC --> DAO["DAO\n(JpaRepository)\ncompartido"]
    IQ --> DAO
```

**Flujo de un comando (escritura):**
1. `POST /api/actividades` → `ActividadComandoControlador` (con `@PreAuthorize`)
2. → `GuardarActividadManejador.ejecutar(ActividadComando)` (abre transacción)
3. → `GuardarActividadUseCase.ejecutar(Actividad)` (valida reglas, autorización contextual)
4. → `ActividadRepositorioComando.guardar(Actividad)` (puerto)
5. → `ActividadRepositorioComandoImplementacion.guardar()` (mapea a entidad JPA, persiste)
6. → Retorna `ComandoRespuesta<UUID>` con el identificador generado

**Flujo de una consulta (lectura):**
1. `GET /api/actividades/area/{id}` → `ActividadConsultaControlador` (con `@PreAuthorize`)
2. → `ConsultarActividadesPorAreaManejador.ejecutar(UUID)` (transacción readOnly)
3. → `ConsultarActividadesPorAreaUseCase.ejecutar(UUID)` (autorización contextual)
4. → `ActividadRepositorioConsulta.consultarPorArea(UUID)` (puerto)
5. → `ActividadRepositorioConsultaImplementacion` (mapea entidades a DTOs)
6. → Retorna `List<ActividadDTO>`

### 5.3 Cableado de Dependencias

Los use cases y servicios de dominio **no son Spring Beans**. Son POJOs instanciados explícitamente mediante clases `@Configuration`:

- **`UseCaseBean.java`** — Declara cada use case como `@Bean` con inyección explícita de puertos por constructor.
- **`ServiceBean.java`** — Declara cada servicio de dominio como `@Bean`.
- **`EncriptarClaveBean.java`** — Declara el `BCryptPasswordEncoder`.

Esto garantiza que el paquete `dominio` no tiene ninguna dependencia de Spring.

### 5.5 Manejo de Excepciones

#### Capa de Dominio — Excepciones Propias

| Excepción | Propósito | HTTP Status |
|-----------|-----------|:-----------:|
| `ValorObligatorioExcepcion` | Campo requerido nulo o vacío | 400 |
| `LongitudExcepcion` | Texto fuera de rango permitido | 400 |
| `PatronExcepcion` | Formato inválido (regex) | 400 |
| `ValorDuplicadoExcepcion` | Violación de unicidad de negocio | 400 |
| `ValorInvalidoExcepcion` | Valor no permitido por regla de negocio | 400 |
| `TiempoVencidoExcepcion` | OTP expirado | 400 |
| `TecnicoExcepcion` | Error técnico inesperado | 500 |

#### Capa de Infraestructura — Manejo Global

- **`ManejadorError`** (`@ControllerAdvice`): Utiliza un `ConcurrentHashMap<String, Integer>` que mapea nombre de excepción → código HTTP. Retorna `Error { nombreExcepcion, mensaje }` como JSON.
- **`ExceptionFilter`** (filtro de seguridad): Captura excepciones del filter chain — `BadCredentialsException` → 400, `AuthorizationException` → 401, otras → 500.

### 6.4 Gestión de Estado

SIBE **no usa NgRx**. El estado se gestiona mediante un `StateService` centralizado basado en `BehaviorSubject<Map<StateProps, any>>`:

```typescript
// Patrón de estado
StateService (providedIn: 'root')
├── stateProps: Map<StateProps, any>
├── state: BehaviorSubject<Map>
├── updateState(prop, value)     // Mutación + push
├── getState(prop)               // Lectura síncrona
├── select<T>(prop): Observable  // Stream reactivo con distinctUntilChanged
├── deleteProperty(prop)         // Eliminación + push
└── rehydrateSession()           // Rehidratación desde sessionStorage al construir
```

**Estado actual:** Un solo key — `StateProps.USER_SESSION` — que almacena `UserSession { correo, identificador, authorities[], logged, rol, direccionId, areaId, subareaId }`.

**Persistencia:** `sessionStorage` (se destruye al cerrar la pestaña — seguridad implícita). En recarga de página, `rehydrateSession()` decodifica el JWT almacenado y reconstruye la sesión.

### 6.5 Comunicación HTTP

Todos los servicios de dominio extienden `HttpService` (Core), que envuelve `HttpClient` con métodos tipados:

| Método | Propósito |

| `doGetById<T>(url, id)` | GET por UUID |

### 7.2 Patrón de Snapshot Temporal

SIBE implementa un **patrón de snapshot temporal** mediante la dualidad Miembro/Participante:

- **`Miembro`** representa los datos maestros actuales del individuo (actualizados semestralmente vía carga masiva).
- **`Participante`** es un snapshot inmutable que captura el estado del miembro en el momento del registro de asistencia.
- **`RegistroAsistencia`** vincula un `Participante` con una `EjecuciónActividad` y es **inmutable** — nunca se actualiza ni elimina.

Esto garantiza que cambios posteriores en los datos maestros (ej. un estudiante cambia de programa académico) no alteren el historial de asistencia.

| `Participante` → `ParticipanteEstudiante` / `ParticipanteEmpleado` / `ParticipanteExterno` / `ParticipanteInterno` | `JOINED` | `participante`, `participante_estudiante`, etc. |

### 8.2 Autenticación — Flujo de Login

```mermaid
sequenceDiagram
    participant FE as Frontend (Login)
    participant RVF as RequestValidationBeforeFilter
    participant AP as UsernamePwdAuthenticationProvider
    participant JTG as JWTTokenGeneratorFilter

    FE->>RVF: Basic Auth (correo:clave en Base64)
    RVF->>AP: Credenciales decodificadas
    AP->>AP: 1. Busca UsuarioEntidad por correo
    AP->>AP: 2. Verifica BCrypt(clave)
    AP->>AP: 3. Verifica estaActivo == true
    AP->>AP: 4. Construye Authentication con identificador
    AP->>JTG: Authentication válida
    JTG->>JTG: Genera JWT con claims:
    Note right of JTG: email, id, authorities, rol,<br/>direccionId, areaId, subareaId<br/>Firma: HMAC-SHA256<br/>Expiración: ~8.33 horas
    JTG->>FE: HTTP 200 + JWT en header Authorization
    FE->>FE: Almacena JWT en sessionStorage
```

### 8.3 Claims del JWT

| `rol` | String | Código del rol (ej. `ADMINISTRADOR_AREA`) |
| `direccionId` | UUID | ID de la dirección asignada (si aplica) |
| `areaId` | UUID | ID del área asignada (si aplica) |
| `subareaId` | UUID | ID de la subárea asignada (si aplica) |
| `exp` | Long | Timestamp de expiración (30,000,000 ms desde emisión) |

## 10. Máquinas de Estados

### 10.1 EjecuciónActividad — Ciclo de Vida de Ejecución

```mermaid
stateDiagram-v2

    PENDIENTE --> EN_CURSO : IniciarActividadUseCase\nhoraInicio = now()
    EN_CURSO --> PENDIENTE : CancelarActividadUseCase\nRevertir a PENDIENTE
```

**Reglas:**
- FINALIZADA es **terminal e irreversible**. Los registros de asistencia son inmutables una vez creados.
- La cancelación revierte a PENDIENTE pero **no resetea** `horaInicio` (valor residual conocido).
- Solo ejecuciones PENDIENTE pueden ser editadas.

**Efectos del estado INACTIVO:**
- No puede autenticarse (filtrado en `UsernamePwdAuthenticationProvider`).

## 13. Decisiones Arquitectónicas (ADRs)

### DA-01 — Arquitectura Hexagonal con Puertos y Adaptadores

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Backend en 3 capas: `dominio` (modelos, puertos, use cases, reglas) → `aplicación` (handlers, fábricas, comandos) → `infraestructura` (controladores, DAOs, seguridad, config) |
| **Alternativas descartadas** | Arquitectura tradicional por capas (acoplamiento dominio-infra); Clean Architecture pura (ceremonia excesiva para el alcance) |
| **Medida de éxito** | 0 imports de Spring/JPA en paquetes `dominio` |
| **Drivers** | AC-07 Mantenibilidad, RT-11, DN-10 |

### DA-02 — CQRS Estructural en Todas las Capas

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Separar controladores, handlers, puertos y use cases en paquetes `comando` y `consulta` en cada capa |
| **Alternativas descartadas** | Controlador único por recurso; CQRS con event sourcing (complejidad innecesaria) |
| **Drivers** | AC-08, RT-12 |

### DA-03 — Transacciones Declarativas en Interfaces de Handler

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `@Transactional` en la interfaz `ManejadorComandoRespuesta<C,R>.ejecutar()`; `@Transactional(readOnly=true)` en interfaces de consulta |
| **Alternativas descartadas** | `@Transactional` en use cases (acoplaría dominio a Spring); en repositorios (granularidad insuficiente) |
| **Drivers** | AC-12, AC-07, RT-13 |

### DA-04 — Motor de Validación por Operación (MotoresFabrica)

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Centralizar reglas de validación en motores estáticos pre-construidos en `MotoresFabrica`, ejecutados por `TipoOperacion` CREAR/MODIFICAR |
| **Alternativas descartadas** | Spring Validation en controladores; validación ad-hoc por use case |
| **Motores** | 13 motores (uno por entidad principal) con 28 implementaciones de `Regla` |
| **Drivers** | AC-07, AC-06 |

### DA-05 — Generación de UUID con Loop Anti-Colisión

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `UtilUUID.generar(Predicate<UUID>)` verifica existencia en repositorio antes de asignar |
| **Drivers** | AC-07, AC-11 |

### DA-06 — Seguridad Stateless con JWT + HTTP Basic

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Autenticación inicial vía HTTP Basic; JWT firmado con HMAC-SHA256 conteniendo claims de identidad, rol y contexto organizacional |
| **Alternativas descartadas** | Sesiones servidor; OAuth2/OIDC (infraestructura excesiva) |
| **Claims** | email, id, authorities, rol, direccionId, areaId, subareaId |
| **Expiración** | ~8.33 horas |
| **Drivers** | AC-04, RT-09, RT-10 |

### DA-07 — Autorización de Doble Capa (Rol + Contexto Organizacional)

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Capa 1: `@PreAuthorize` con SpEL en controladores; Capa 2: `AutorizacionContextoOrganizacionalServicio` en use cases verificando pertenencia organizacional |
| **Alternativas descartadas** | RBAC simple; Spring Security ACL; Row-level security en BD |
| **Drivers** | AC-05, AC-14 |

### DA-08 — Eliminación Lógica de Usuarios (Soft Delete)

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | Campo `estaActivo` booleano; usuarios inactivos filtrados en todas las consultas |
| **Efecto** | Preservación del historial de actividades asociadas |
| **Drivers** | AC-11 |

### DA-09 — Datos de Referencia Sembrados por DataLoaders

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `CommandLineRunner` con `@Order` para pre-poblar datos de referencia; `hibernate.ddl-auto=update` |
| **Alternativas descartadas** | Flyway/Liquibase (no implementado); SQL scripts manuales; `data.sql` |
| **Drivers** | RT-05, DN-10 |

### DA-10 — Frontend Core/Shared/Feature con Lazy Loading

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `CoreModule` (singleton, guards, interceptors, layout); `SharedModule` (componentes reutilizables); Feature modules (lazy-loaded por ruta) |
| **Regla** | Feature → Core/Shared ✅; Feature → Feature ❌ |
| **Profundidad** | Hasta 3 niveles de lazy loading (App → Home → Area → SubArea) |
| **Drivers** | AC-09, AC-01, RT-04 |

### DA-11 — Estado Frontend con BehaviorSubject + sessionStorage

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `StateService` centralizado con `BehaviorSubject`; rehidratación desde `sessionStorage`; sin NgRx |
| **Seguridad** | `sessionStorage` se destruye al cerrar pestaña |
| **Drivers** | AC-04, DN-06 |

### DA-12 — Interceptores HTTP Duales en Frontend

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `AuthInterceptor` (inyecta credenciales Basic o Bearer JWT); `TokenInterceptor` (verifica expiración, fuerza logout) |
| **Drivers** | AC-04, DA-06 |

### DA-13 — Motor de Procesamiento Excel Reutilizable

| Aspecto | Detalle |
|---------|---------|
| **Decisión** | `ProcesadorExcelMotor` genérico con `FilaExcelMapeador<T>`; implementaciones específicas para estudiantes y empleados |
| **Drivers** | AC-13, RT-14 |

### DA-14 — Manejo Centralizado de Excepciones

| Aspecto | Detalle |

| **Drivers** | AC-07, AC-06 |

### DA-15 — CORS Restringido a Origen Conocido

| Aspecto | Detalle |

| Aspecto | Detalle |

| **Drivers** | AC-01, AC-05 |

### DA-18 — Chart.js para Dashboards + xlsx para Exportación Excel en Cliente

| Aspecto | Detalle |

| **Decisión** | Chart.js 4.5.0 para visualización; xlsx 0.18.5 para generación de Excel sin roundtrip al backend |
